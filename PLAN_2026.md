### 1B. UI/UX polish — 6–10 days

- Verify every change in light/dark/race + mobile + RTL (hard rule).

### 1C. Translation completeness — 4–7 days
- Audit `src/lib/i18n.ts` (en/fr/ar, ~154 keys each) + `src/components/coach/copy.ts` for gaps/desync. Write a tiny key-parity check script (en vs fr vs ar) and wire into `lint`/CI.
- Fill missing FR/AR copy across public + organizer pages (admin stays EN by design).
- **Persist selected locale** beyond query strings (cookie + `<html lang dir>`), default to FR/AR for Algeria via `Accept-Language`.
- RTL polish: nav order, icon positions, button/dropdown alignment, form rhythm.

**WS1 subtotal: ~12–21 days (~2.5–4 weeks).**

## WS2 — AI Coach: richer onboarding & per-user context

Goal: capture a real runner profile so the model has stronger grounding. New onboarding data:
running history, typical/peak volume, target (race + time), **injury history**, **chronic conditions/diseases**.

### Steps
1. **Data model** (1–2 d): extend `RunnerGoal` (or add `RunnerProfile`) — `historySummary`, `yearsRunning`, `typicalWeeklyKm`, `peakWeeklyKm`, `recentRaces[]`/PBs, `injuryHistory[]` (area, date, status), `chronicConditions[]` (tagged), `medications?`, `restingHR?`. Prisma migration. (Note: coach uses raw SQL — follow existing pattern in `service.ts`.)
2. **Schemas + form** (2–3 d): extend `src/lib/coach/schemas.ts` Zod + `coach-goal-form.tsx` into a multi-step onboarding (basics → history/volume → target → health). Add EN/FR/AR copy in `copy.ts`.
3. **Context builder** (1–2 d): fold new fields into `context.ts` compactly (summaries, not raw dumps — protects token cost & quality). Keep PII out.
4. **Safety — important** (2–3 d): chronic conditions (cardiac, asthma, diabetes, hypertension…) must gate advice. Extend `safety.ts`: condition-aware cautions, escalate "see a professional", never prescribe. **Have a sports-health professional review** (already an open TODO item).
5. **Plan personalization** (1–2 d): let deterministic planner use volume/history for safer starting load; model explains.
6. **Tests** (1–2 d): extend `test:coach` fixtures for beginner/injured/chronic-condition runners; authz + output-schema checks.

**WS2 subtotal: ~8–14 days (~1.5–3 weeks).** Plus external coach/medical review (calendar time, not dev time).

## WS3 — Android + iOS apps

App is SSR + server-actions heavy → a **native/React Native rewrite is expensive and duplicative**.
Recommended path is staged:

### Phase A — PWA foundation (3–5 days) — do this regardless
- Add `manifest.json` (icons from WS1A), installability, offline shell, ensure FCM SW coexists.
- Mobile viewport/safe-area/tap-target audit. Ships value immediately (installable on Android, partial iOS).

### Phase B — Capacitor wrapper (8–14 days) — recommended for store apps
- Wrap the deployed web app in **Capacitor** (Android + iOS shells), point at hosted ZidRun.
- Native bridges: push (FCM/APNs), camera/photo picker for uploads, deep links, status bar/splash, secure session storage.
- App icons/splash, store metadata, screenshots, privacy policy.
- Build pipeline; **Apple Developer ($99/yr) + Google Play ($25 once)** accounts; iOS needs a Mac/CI (e.g. Codemagic/EAS) for builds.
- Risk: Auth.js cookie/session flow + redirects inside a webview need testing; OAuth (Google) in-app browser handling.

### Phase C — Store submission & review (calendar 1–3 weeks, ~3–5 dev days)
- First Apple review is the wildcard (rejections common for "just a website" wrappers — mitigate with native push/camera/offline so it's app-like).

**WS3 subtotal: ~14–24 dev days (~3–5 weeks) + store-review calendar time.**
*(A full React Native rewrite instead would be ~8–14 weeks — not recommended now.)*
